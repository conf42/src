Who broke the build? — Using Kuttl to improve E2E testing and release faster

Video size:

Abstract

No one wants to be responsible for breaking the build. But what can you do as a developer to avoid being the bad guy? How can project leads enable their teams to reduce the occurrence of broken builds?

Summary

  • Ram Mohan Rao Chuka talks on who broke the bell using Kuttl to improve end to end testing and release faster. How can project leads enable their teams to reduce the occurrence of broken bells?
  • Do software updates really matter? In the current world, software is run on everywhere. JFrog mission is to power all the software upgrades in the world. The company has moved from a waterfall model to a daily release cycle in DevOps.
  • In this session I will cover a quick history of our testing challenges and what led us to KutTl. It's more like a single click setup where everything is automated, you develop and test locally. This helps to save us time and to reproduce production issues at ease.
  • Most developers don't write end to end teams and these teams are actually executed on a remote CI series server. Kuttl is a toolkit for writing teams, mainly designed for Kubernetes operators. Instead of running these tests remotely you can run them locally. This would cut down the headaches and valuable time wasted.
  • Kuttl is both a CLI tool as well as an API for testing. It allows you to test kubernetes applications without writing any go code. Kuttl also provides a cloud native way of doing testing. Here's a demo of how to use kuttl.
  • Kuttl is an open source tool anybody can contribute and it's free to use. It can be used for local end to end testing. It provides detailed output logging onto the console. Can also be integrated with the CI pipeline to see how the test actually failed.
