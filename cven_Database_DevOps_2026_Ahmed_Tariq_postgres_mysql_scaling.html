Schema Migrations in CI/CD: When AI Helps, When It Hurts, and How to Tell the Difference

Video size:

Abstract

Schema migrations are the riskiest deploy most teams ship. Application releases can be rolled forward and back. A migration that locks a 2TB table for an hour, or quietly loses data on rollback, cannot. And now AI wants to write them for us. This talk is a working CI/CD field guide for shipping database changes safely in cloud environments, and an honest assessment of where AI-assisted review fits inside that pipeline. We’ll cover the migration patterns that hold up: backward-compatible schema changes, expand-and-contract releases, automated migration tests, approval gates, and post-deploy verification. Then we’ll add the AI layer where it earns its keep, including drafting forward and rollback DDL, flagging unsafe operations before review (full-table rewrites, locks held in transactions, missing concurrency hints), and triaging incidents when a migration goes sideways. I’ll also be direct about where AI breaks down. Hallucinated database syntax, like Postgres flags that are actually MySQL flags. Missing the operational context that a “trivial” migration on a hot table will lock production for an hour. Rollback scripts that compile but quietly lose rows. The framework, three axes for scoring any migration workflow on blast radius, reversibility, and human-in-loop requirement, gives engineers a defensible way to decide where automation and AI belong in their pipeline, and where they don’t. Attendees leave with concrete CI/CD patterns they can adopt next sprint, the specific failure modes to design around, and a practical decision rule for keeping engineering judgement in control of production data.
