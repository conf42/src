K8s Gateway API 101

Video size:

Abstract

As the Gateway API recently reached the stable level, we should switch from Ingresses to a more efficient way to route the requests to our services. We will delve into this role-oriented, expressive, and extensible project, with some examples and a live demo, comparing it with other major solutions

Summary

  • Gregorio Palama talks about Kubernetes gateway API. We are talking about exposing our services outside the cluster. The important thing is the way that the requests is routed.
  • Ingress exposes HTTP and HTTPs routes from outside the cluster to services within the cluster. In traffic routing is controlled by rules defined on the ingress results. Everything is defined in the same resource. There are other limitations, for example extensibility. It relies on annotations.
  • The Gateway API is a global available since Kubernetes 125 that has been released in October. API Gateway describes anything that exposes capabilities of a backend service while providing extra capabilities for traffic routing and manipulation.
  • Gateway API defines a set of gateways that share a common configuration and behavior. Routes define protocol specific rules for mapping requests from a gateway to a Kubernetes services. The possibility to be attached can be limited and filters and advanced rules can be applied on the route.
  • For the demo I will use a cluster that is served by Google cloud platform. Using Google Kubernetes engine, we have different gateway class that are served by the platform. The interesting stuff is that we have two routes for the same service. And that's something that we can think about in the real world.
  • Gum stands for Gateway API for mesh management and administration. It is an initiative that allows the API to works not only north south but east west. A route can be associated directly with a service too.
  • The ingresses API has an insufficient permission model. The Gateway API permission model with the three personas is definitely better than the Ingress API one. So that's why we should start using the Gateway API instead of Ingress. I hope you will switch to the gateway API very soon.
