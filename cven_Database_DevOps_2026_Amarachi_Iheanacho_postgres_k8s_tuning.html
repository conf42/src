Can a Kubernetes Operator Replace a DBA? A 30-Day Experiment

Video size:

Abstract

Kubernetes operators promise to automate complex operational tasks, backups, failovers, upgrades, scaling, and turn database management into a declarative workflow. But how far does that promise actually go? In this talk, I run a 30-day experiment to find out. Using a PostgreSQL cluster deployed with the CloudNativePG operator, I handed over day-to-day database operations to Kubernetes and observed what happened in real-world conditions. Over the course of a month, I monitored the system while intentionally introducing common production problems: node failures, pod crashes, storage pressure, replication issues, upgrades, and backup recovery scenarios. Some problems were resolved automatically within seconds. Others exposed surprising limitations where manual intervention was still required. This talk walks through the experiment week by week, showing what the operator handled gracefully, what broke, and where human operators are still essential. Along the way we’ll explore the operational model behind Kubernetes operators, the trade-offs of automated database management, and the practical lessons teams should know before trusting automation with their stateful workloads. If you’ve ever wondered whether operators can truly replace traditional database administration, or simply shift the role, this experiment provides a candid, data-driven answer. Key Takeaways Attendees will learn: - How Kubernetes operators manage stateful systems like PostgreSQL - Which database failures operators can automatically recover from - Where manual operational knowledge is still required - The operational trade-offs of running databases in Kubernetes - Practical lessons from a month-long real-world experiment
